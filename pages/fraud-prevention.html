07 · HMRC-specific control area

Fraud prevention data

How software that uses VAT MTD or Income Tax Self Assessment MTD APIs captures and validates the fraud prevention headers HMRC requires.

Header dataValidationMTD APIs

Questions this page answers

Do you submit fraud prevention data where required?

Relevant HMRC APIs expect header data that helps identify the request environment and reduce fraud risk.

Have you validated the header implementation?

Before production use, the business should test that headers are correctly formatted and transmitted.

How do you keep the data accurate?

The application should minimise spoofing, document the source of header values and monitor implementation changes.

Implementation posture

Where Zody AI Ltd uses HMRC APIs that require fraud prevention data, the software is expected to collect and transmit the necessary header information in accordance with HMRC’s specification. That includes validating the format of header values, checking end-to-end behaviour before live use and keeping the implementation under review as the client and backend stack evolve.

  • Fraud prevention headers only where the API requires them.
  • Controlled collection of device, application and request context.
  • Testing against HMRC’s fraud prevention validation path before go-live.
  • Review processes when mobile, web or proxy behaviour changes.

Practical note

If your current product does not yet submit to VAT MTD or ITSA MTD APIs, keep this page, but state that the controls are planned and will be enforced before that integration goes live.
